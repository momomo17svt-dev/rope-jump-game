# 【要件定義・基本仕様書】大縄跳びサバイバル

> **AIエージェントへの指示:** この仕様書を最初に読み込み、記載された内容に従って実装を進めること。不明点は質問すること。

---

## 1. プロジェクト概要

- **アプリ名:** 大縄跳びサバイバル（仮）
- **ジャンル:** カジュアル・エンドレスアクションゲーム
- **コンセプト:** 迫りくる大縄をタイミングよくタップで跳び続け、己の限界（連続ジャンプ回数）に挑む。
- **デザイン:** シンプルな2D。キャラクター・縄・背景はすべてコード（React Native View / SVG）で描画する。外部画像アセットは不要。`assets/` フォルダは後から差し替え可能なプレースホルダー構成とすること。
- **プレイモード:** シングルプレイ専用（通信対戦なし）。自身のスコアをグローバルランキングに送信して競う。

オフラインでも快適に遊べる「ローカルファースト」設計とし、オンライン時にランキングを同期するハイブリッド構成を採用する。

## 3. 画面構成と機能仕様

### ⓪ 初回起動ダイアログ

アプリ起動後、SQLiteの `local_user` テーブルにレコードが存在しない場合のみ表示する。

- **UI要素:**
  - 「プレイヤー名を入力してください」のテキストフィールド（1〜12文字）。
  - 「決定」ボタン。
- **処理:**
  - `expo-crypto` 等でUUIDを生成し `device_id` とする。
  - `user_name` と `device_id` を `local_user` テーブルに保存。
  - 完了後、タイトル画面へ遷移する。

### ① タイトル画面

- **UI要素:**
  - ゲームタイトルロゴ。
  - 「TAP TO START」ボタン（画面全体をタップ領域としても可）。
  - 「ランキングを見る」ボタン。
  - 画面下部に、SQLiteから取得した「現在の自己ベスト（ハイスコア）」を表示。

### ② ゲーム画面

- **レイアウト:**
  - 左右両端: 縄を回すキャラクター（自動アニメーション）。
  - 中央: プレイヤーキャラクター。
  - 上部中央: 現在の「ジャンプ成功回数」を大きく表示。
- **操作とアクション:**
  - 画面内の任意の場所をタップするとキャラクターがジャンプする。
  - ジャンプは「上昇〜下降〜着地」の一定フレームのアニメーションで構成される。
- **縄の挙動と難易度:**
  - 開始時に「3, 2, 1, GO!」のカウントダウン。
  - 縄は一定周期で回転する。
  - スコア（ジャンプ回数）が10増えるごとに、縄の回転速度が5%上昇する。
- **当たり判定とゲームオーバー:**
  - 縄が「最下点（キャラクターの足元）」を通過する瞬間に、キャラクターのY座標が「安全な高さ（ジャンプ中）」にない場合、衝突と判定してゲームオーバーとする。

### ③ リザルト画面

- **UI要素:**
  - 「今回のスコア」を表示。
  - SQLiteの自己ベストと比較し、更新していれば「NEW RECORD!」の特別な演出を表示。
  - 「もう一度遊ぶ（リトライ）」「タイトルへ戻る」ボタン。
- **バックグラウンド処理（ランキング同期）:**
  - 自己ベストを更新した場合、端末がオンラインであればバックエンドのランキングAPI（POST）へスコアを非同期で送信する。

### ④ ランキング画面

- **UI要素:**
  - バックエンドAPI（GET）から取得したグローバルランキング（上位100名）をリスト表示（順位、ユーザー名、スコア）。
  - `device_id` で自分のエントリーを識別し、自分の順位とスコアを画面下部にハイライト表示。
  - 「戻る」ボタン。

## 4. データモデル設計

### 4.1 ローカルデータベース（Expo SQLite）

#### `local_user` テーブル（プレイヤー情報）

| カラム名 | データ型 | 制約・説明 |
| :--- | :--- | :--- |
| `id` | INTEGER | PRIMARY KEY AUTOINCREMENT |
| `device_id` | TEXT | UUID（端末固有。`expo-crypto` 等で生成） |
| `user_name` | TEXT | プレイヤー名（1〜12文字） |

*※レコードは常に1件のみ。存在チェックで初回起動ダイアログの表示を制御する。*

#### `local_scores` テーブル（プレイ履歴）

| カラム名 | データ型 | 制約・説明 |
| :--- | :--- | :--- |
| `id` | INTEGER | PRIMARY KEY AUTOINCREMENT |
| `score` | INTEGER | ジャンプ成功回数 |
| `played_at` | TEXT | プレイ日時（ISO 8601形式） |

*※`MAX(score)` を自己ベストとして扱う。*

### 4.2 バックエンドAPIペイロード

**`POST /api/scores`** — スコア送信・Upsert（device_idが同一なら上書き）

```json
{
  "device_id": "端末固有のUUID文字列",
  "user_name": "プレイヤーの登録名",
  "score": 150
}
```

**`GET /api/rankings`** — グローバルランキング取得（上位100件）

```json
[
  { "rank": 1, "user_name": "Alice", "score": 300 },
  { "rank": 2, "user_name": "Bob",   "score": 250 }
]
```

## 5. 開発フェーズ（ステップ）

AIエージェントは以下のフェーズに従い、段階的に実装を進めること。**各フェーズ完了後に動作確認を行ってから次フェーズに進む。**

### Phase 1: フロントエンド基盤とローカルロジック（完全オフライン）

1. Expoプロジェクトのセットアップと横画面固定の設定。
2. `expo-crypto` を用いた `device_id` 生成と、初回起動ダイアログの実装。
3. タイトル・ゲーム・リザルト・ランキング画面のルーティング実装。
4. 大縄跳びのコアロジック実装:
   - 縄の回転アニメーション（コード描画）
   - タップジャンプ（上昇・下降・着地）
   - 当たり判定
   - スコア10ごとの難易度上昇（回転速度+5%）
5. Expo SQLiteのセットアップ（`local_user` / `local_scores` テーブル）。
6. スコアのローカル保存・自己ベスト表示機能の実装。

### Phase 2: バックエンドAPIの構築

1. Go言語とDockerを用いたAPIサーバーの構築。
2. `render.yaml` の作成（Web Service のみ。DB は Neon を別途使用）。
3. PostgreSQLのスキーマ定義（`global_rankings` テーブル）とDB接続。
4. スコアのUpsert処理（`POST /api/scores`）の実装。
5. ランキング取得処理（`GET /api/rankings`）の実装。

### Phase 3: フロントエンドとAPIの統合

1. ランキング画面のUI実装（`device_id` によるハイライト表示を含む）。
2. リザルト画面での自己ベスト更新時のAPIスコア送信処理の組み込み（非同期・オンライン時のみ）。
